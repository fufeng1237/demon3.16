# 内河船舶任务分配与自适应重分配系统 — 技术文档

## 1. 系统概述

面向内河运输场景的多船多任务调度系统。给定 M 艘船和 K 个运输任务（装货港→卸货港），最小化船队总航行代价，并在运行过程中实时响应异常事件进行自适应重分配。

## 2. 路网构建

### 2.1 输入

- **地图**: `binary_map_scaled.png` (4571×1626 像素, 黑=陆地, 白=水域)
- **像素比例**: 2.0 m/pixel, 覆盖面积 9.1km × 3.3km

### 2.2 算法流程

```
Step 1: 加载 → cv::threshold(127) → 二值图 (黑白)
Step 2: 骨架提取 → cv::ximgproc::thinning (Guo-Hall 算法)
Step 3: 关键像素 → 8邻域度数 ≠ 2 的像素 (端点 + 交叉点)
Step 4: DBSCAN 聚类 → eps=25像素, 合并邻近关键点
Step 5: 路径追踪 → 从关键点沿骨架走到下一关键点
Step 6: 构建图 → 聚类中心为节点, 聚类间最短路径为边
Step 7: 锚点注入 → 港口/船舶/加油站连接到最近节点
Step 8: 两阶段剪枝 → 孤立节点 + 无锚点死胡同
Step 9: Floyd-Warshall → 全源最短路径距离矩阵
```

### 2.3 关键算法细节

**Guo-Hall 骨架提取** (OpenCV `cv::ximgproc::THINNING_GUOHALL`):
- 迭代删除边界像素, 保持拓扑连通性
- 输出 1 像素宽的中心线

**DBSCAN 聚类**:
- `eps = 25` 像素 (50m), 将相邻的端点/交叉点合并
- 聚类中心 = 成员像素坐标的算术平均
- 目的: 减少节点数量, 消除骨架碎片化

**路径追踪** (`trace_paths_between_clusters`):
- 从每个关键像素出发, 沿骨架走到下一个关键像素
- 只经过度数为 2 的路径像素
- 记录聚类间的最短路径长度

**两阶段剪枝**:
- Phase 1: 移除度数为 0 的孤立节点
- Phase 2: 移除度数为 1 且邻居也是普通节点的死胡同 (保留连接锚点的分支)

### 2.4 图数据结构

```cpp
// C++ 侧 (对标 usv_roadmap_standalone)
struct GraphNode {
    int id;
    int x, y;           // 像素坐标
    string type;        // "node" | "port" | "ship" | "gas_station"

```

## 3. 任务分配

### 3.1 数据建模

```
Task (任务):
  - pickup_node:      装货港路网节点ID
  - delivery_node:    卸货港路网节点ID
  - payload:          货物重量 (吨)
  - priority:         优先级 1/2/3
  - deadline:         截止时间 (秒)
  - status:           PENDING → ASSIGNED → LOADING → TRANSPORTING → FINISHED
  - dynamic_priority: 动态优先级 (随时间变化)
  - downstream_tasks: 依赖此任务的后续任务列表
  - upstream_task:    此任务依赖的前置任务

Ship (船舶):
  - max_payload:      最大载重 (吨)
  - max_energy:       最大能源 (kWh)
  - max_speed:        最大航速 (m/s)
  - current_node:     当前所在路网节点
  - energy:           当前能源
  - load:             当前载货
  - health:           健康度 0~1
  - task_sequence:    已分配的任务ID列表 (按执行顺序)
  - current_phase:    当前阶段 (navigate_to_pickup/loading/navigate_to_delivery/unloading)
```

### 3.2 初始分配算法

**贪心分配** (`greedy_assign`):

```
1. 按紧急度排序任务: urgency = priority × 10 + 100 / deadline
2. 对每个任务:
   a. 遍历所有船, 检查约束:
      - payload ≤ ship.remaining_capacity
      - dist_matrix[ship_node, pickup_node] + dist_matrix[pickup, delivery] < ∞
      - 预估能耗 ≤ ship.energy × 0.7
   b. 分配代价 = 航行距离 + 时间紧急度惩罚
   c. 选择代价最小的船
3. 返回分配结果
```

**2-opt 优化** (`two_opt_optimize`):
- 对每艘船的任务序列, 尝试反转子序列
- 如果总路径长度减少 → 接受交换
- 迭代至收敛 (最多 100 轮)

**船间转移** (`optimize_task_transfer`):
- 尝试将任务从船A转移到船B
- 检查船B的载重约束
- 如果船队总代价降低 → 接受转移

### 3.3 代价函数

```python
Cost(ship, task) = d_pickup + d_exec                # 航行距离 (路网最短路径)
                 + time_urgency × task.priority      # 时间紧急度
                 + energy_risk × 10                  # 能源风险 (超过50%余量)
                 + load_penalty                      # 负载均衡惩罚 (>80%满载)
```

### 3.4 输出格式

```
Ship_0: T1 → T3
  Step 1: Navigate Port_J → Port_J (load 800t, 0.0km)
  Step 2: Navigate Port_J → Port_H (unload 800t, 1.0km)
  Step 3: Navigate Port_H → Port_J (load 1000t, 1.0km)
  Step 4: Navigate Port_J → Port_F (unload 1000t, 0.4km)
  总航程: 2.3km, 能耗: 5.9kWh, 耗时: 0.1h
```

## 4. 实时自适应调度

### 4.1 调度循环

```
每个周期 (默认60秒):
  1. advance_time(dt)           # 更新所有船的位置/能源/任务进度
  2. update_priority()          # 更新所有任务动态优先级
  3. detect_events()            # 检测异常事件
  4. handle_events()            # 处理事件 (含失败回滚)
  5. reassign_tasks()           # 重分配释放的任务
  6. update_visualization()     # 刷新 RViz Marker
```

### 4.2 船状态更新

```
状态机: PENDING → NAVIGATE_TO_PICKUP → LOADING → NAVIGATE_TO_DELIVERY → UNLOADING → (下一任务)

每周期更新:
  - 航行中: 按速度 × dt 推进距离, 累加能耗, 到达时切换阶段
  - 装货中: 等待 loading_time (5分钟), 完成后 ship.load += payload
  - 卸货中: 等待 unloading_time (3分钟), 完成后 ship.load -= payload, 标记任务完成
```

### 4.3 事件检测 (6 种)

| 事件 | 触发条件 | 检测方式 |
|------|---------|---------|
| energy_low | energy < 25% max | 硬阈值 |
| energy_critical | energy < 15% max | 硬阈值 |
| fault | health < 30% | 硬阈值 |
| task_overdue | current_time > deadline | 时间比较 |
| ship_idle | 任务序列为空 + 有待分配任务 | 状态检查 |
| priority_spike | dynamic_priority > base × 2.0 | 动态优先级对比 |
| energy_trend | 能耗趋势预测不足 | 历史能量斜率分析 |

### 4.4 重分配处理

**能源不足** (`_handle_energy_low`): 在任务序列前插入"去最近加油站"步骤

**能源紧急** (`_handle_energy_critical`): 释放所有未开始任务, 重新分配给其他船

**故障** (`_handle_fault`): 释放全部任务, 已装货任务由 `TaskFailureHandler` 处理

**任务延期** (`_handle_overdue_task`): 转移给最快可用船

**优先级剧变** (`_handle_priority_spike`): 触发 overdue 处理逻辑

## 5. 动态优先级

### 5.1 公式

```
P(t) = (base_priority + chain_bonus) × time_urgency × (1 - env_penalty)

其中:
  chain_bonus  = 0.5 × len(downstream_tasks)    # 下游有等待任务 → +0.5/task
  time_urgency = 1.0 + 5.0 × (3600 / remaining) # 1小时内到期 → 最高6倍
  env_penalty  = 0.5 (ship_energy < 20%)        # 船能源不足 → 降50%
                 0.3 (ship_health < 50%)         # 船健康差 → 降30%
```

### 5.2 任务链依赖

```
同艘船的任务序列: T5 → T3 → T2
  T5.downstream = [T3], T3.downstream = [T2]
  T3.upstream = T5, T2.upstream = T3

作用: T5 延期 → T3 优先级提高 → T2 优先级也提高 (级联效应)
```

## 6. 任务失败回滚

### 6.1 按阶段回滚策略

| 失败阶段 | 货物状态 | 回滚策略 |
|---------|---------|---------|
| PENDING/ASSIGNED | 未装货 | 直接释放 → 重新分配 |
| NAVIGATE_TO_PICKUP | 未装货 | 释放 → 重新分配 (已消耗航程沉没) |
| LOADING | 装货中 | 货物回滚到港口 → 释放重分配 |
| NAVIGATE_TO_DELIVERY | **已装货在途** | **就近卸货 + 接力船完成** |
| UNLOADING | 卸货中 | 重试卸货 |

### 6.2 接力机制

```
场景: Ship_0 故障, T5 在途 (已装货 800t, 正在前往 Port_F)

处理:
  1. 找最近港口: Port_G
  2. Ship_0: 转向 Port_G 卸货 (load -= 800t)
  3. 创建接力任务 T_100: Port_G → Port_F (800t, priority+1)
  4. 分配给最快可用船 Ship_2
  5. T5 标记为 cancelled
```

## 9. 需要改进的地方

### 9.1 算法层面

| 问题 | 现状 | 改进方向 |
|------|------|---------|
| **路径规划粗糙** | 路网节点间用 Floyd-Warshall 最短路径, 未调用底层 A* | 集成 `ais_planner_node` 的 A* 路径, 得到真实航线 |
| **直线边切断陆地** | DBSCAN 聚类后用直线连接, 部分边穿过陆地 | 沿骨架路径存储中间点, 画边用折线代替直线 |
| **无排队/拥堵** | 多船同时到同一港口不冲突 | 港口容量队列 + 等待时间建模 |
| **静态 scale_factor** | scale_factor=1.0, 边权重=像素距离 | 用真实物理距离 (像素×2.0m/px) |
| **2-opt 收敛慢** | O(n²) 交换, 迭代上限 100 | 用 LKH 或 OR-Tools 求解器 |

### 9.2 模型层面

| 问题 | 现状 | 改进方向 |
|------|------|---------|
| **纯规则系统** | 贪心 + 2-opt 是启发式, 无学习能力 | 引入 HGT 异构图编码器 + PPO 强化学习 |
| **无预测能力** | 事件检测是阈值触发, 无预测 | 用历史数据训练异常预测模型 |
| **优先级静态** | 虽已支持动态优先级, 但参数是经验值 | 用数据学习 urgency 函数的参数 |
| **代价函数手动** | 各项权重是经验设定 | 多目标优化 Pareto 前沿 + 逆强化学习 |

### 9.3 工程层面

| 问题 | 现状 | 改进方向 |
|------|------|---------|
| **Python ↔ C++ 双重坐标** | C++ 输出像素坐标, Python 转世界坐标 | 统一在 C++ 侧输出世界坐标 |
| **JSON 无距离矩阵** | C++ 不输出距离矩阵, Python 每次重算 | C++ 侧直接输出 Floyd-Warshall 结果 |
| **可视化 B版 hack** | flipud + origin='lower' + fy(world_h - y) | 统一坐标系, 消除翻转 |
| **无 ROS 集成** | 目前是独立 Python 脚本 | 接入 ROS topic/subscribe GPS/电池/任务 |
| **测试数据固定** | 船/任务是硬编码的 | 从 YAML 配置文件读取 |

### 9.4 功能缺失

| 缺失 | 影响 | 优先级 |
|------|------|--------|
| 多船协调 (同一港口排队) | 高并发时分配不实际 | 高 |
| 加油站加油后继续 (实际建模) | 能源管理不完整 | 高 |
| 任务取消/修改 | 任务变更需重启系统 | 中 |
| 天气/水流影响航速 | 实际航行时间偏差大 | 中 |
| 船舶维护调度 | 无预防性维护模型 | 低 |
| 历史数据持久化 | 无法回溯分析 | 低 |
