# 基于异构图与Graph-guided ALNS的多无人船动态协同运输调度系统

---

# 一、系统目标

本系统针对多无人船协同运输问题，实现运输任务的智能分配、持续优化与动态重调度。

系统满足以下要求：

* 所有运输任务初始已知（可新增任务）
* 船舶位置实时变化
* 一艘船可提前规划未来5~10个运输任务
* 一艘船允许同时运输多个货物
* 正常情况下，一个运输任务始终由同一艘船完成
* 出现异常时允许任务转交
* 输出每艘船未来需要访问的路网节点序列（RoadNode Sequence）

---

# 二、整体框架

整个系统采用

> **Graph + Graph-guided ALNS + Rolling Horizon**

整体流程如下：

```text
初始化系统
      │
      ▼
建立Ship-Road-Task异构图
      │
      ▼
HGT学习节点关系
      │
      ▼
Graph-guided ALNS
      │
      ▼
生成每艘船RoadNode访问序列(Route)
      │
──────────────────────────────
      │
船舶执行Route
      │
      ▼
Graph实时更新
      │
      ▼
Rolling Horizon滚动优化
      │
      ▼
状态检测(Event)
      │
      ├──────────────┐
      │              │
     无事件        检测到事件
      │              │
      ▼              ▼
继续执行       Route Repair
      │              │
      └──────┬───────┘
             ▼
       更新Route继续执行
```

整个系统始终维护一张动态异构图。

---

# 三、异构图建模

建立异构图：

```text
G=(V,E)
```

其中

节点：

```text
Ship

RoadNode

Task
```

边：

```text
Ship→RoadNode

RoadNode↔RoadNode

RoadNode→Task

Task→RoadNode

Ship→Task
```

Graph实时维护整个运输系统状态。

---

# 四、节点设计

## （1）Ship节点

表示运输资源。

### 静态属性

| 属性          | 说明    |
| ----------- | ----- |
| ShipID      | 船舶编号  |
| MaxCapacity | 最大载重量 |
| MaxEnergy   | 最大能源  |

### 动态属性

| 属性                | 说明        |
| ----------------- | --------- |
| CurrentRoadNode   | 当前所在节点    |
| CurrentSpeed      | 当前速度      |
| Heading           | 当前航向      |
| RemainingCapacity | 剩余容量      |
| RemainingEnergy   | 剩余能源      |
| ETA               | 当前预计完成时间  |
| QueueLength       | 当前Route长度 |
| WorkingState      | 工作状态      |

---

## （2）RoadNode节点

统一表示：

* 港口
* 航点
* 加油站

属性：

| 属性         | 说明                     |
| ---------- | ---------------------- |
| NodeID     | 节点编号                   |
| Longitude  | 经度                     |
| Latitude   | 纬度                     |
| NodeType   | Port / WayPoint / Fuel |
| Congestion | 当前拥堵程度                 |

RoadNode共同组成整个航路网络。

---

## （3）Task节点

Task表示运输需求。

属性：

| 属性                | 说明     |
| ----------------- | ------ |
| TaskID            | 编号     |
| PickupNode        | 装货节点   |
| DeliveryNode      | 卸货节点   |
| CargoWeight       | 货物重量   |
| CargoVolume       | 货物体积   |
| RemainingDistance | 剩余运输距离 |

Task本身不会执行。

真正执行的是：

```text
PickupNode

↓

DeliveryNode
```

---

# 五、边设计

## （1）Ship→RoadNode

表示船当前位置。

属性：

| 属性       |
| -------- |
| Distance |
| ETA      |

---

## （2）RoadNode↔RoadNode

表示航道连接关系。

属性：

| 属性             |
| -------------- |
| Distance       |
| TravelTime     |
| FuelCost       |
| CongestionCost |

---

## （3）RoadNode→Task

表示装货关系。

```text
RoadNode

↓

Task
```

属性：

```
Relation = Pickup
```

---

## （4）Task→RoadNode

表示卸货关系。

```text
Task

↓

RoadNode
```

属性：

```
Relation = Delivery
```

因此：

```
PickupNode

↓

Task

↓

DeliveryNode
```

即可完整表示运输需求。

---

## （5）Ship→Task

表示Graph预测船执行该任务的综合能力。

属性：

| 属性               | 说明         |
| ---------------- | ---------- |
| MatchingScore    | 综合评分       |
| InsertCost       | 插入Route代价  |
| ETA              | 到达Pickup时间 |
| EnergyCost       | 能耗         |
| CapacityFeasible | 是否满足容量     |

Graph重点学习这类关系。

---

# 六、第一次调度（Initial Scheduling）

第一次调度不是：

```
Task

↓

Ship
```

而是：

**直接生成每艘船未来的RoadNode访问序列(Route)。**

流程如下：

```
建立Graph

↓

HGT编码

↓

计算Ship-Task匹配关系

↓

Graph-guided ALNS

↓

生成Route
```

例如：

```
Ship1

↓

Node15

↓

Node25(Pickup Task3)

↓

Node42(Pickup Task5)

↓

Node87(Delivery Task3)

↓

Node96(Delivery Task5)
```

Graph负责学习关系。

ALNS负责真正优化Route。

---

# 七、Graph-guided ALNS优化

Graph首先计算：

```
Task18

↓

Ship1

0.95

Ship2

0.91

Ship5

0.86
```

保留Top-K候选船。

ALNS仅搜索这些候选船。

然后不断执行：

```
Destroy

↓

Repair

↓

Destroy

↓

Repair
```

不断优化：

* 节点访问顺序
* 多货物装载顺序
* Pickup/Delivery顺序
* 容量约束

最终得到总代价最低的Route。

---

# 八、系统输出

调度系统最终输出：

**每艘船未来需要访问的RoadNode序列。**

例如：

```
Ship1

[15,25,42,87,96]
```

```
Ship2

[8,16,31,54]
```

```
Ship3

[5,28,73,81]
```

系统不负责节点之间如何运动。

仅输出：

RoadNode访问顺序。

---

# 九、执行阶段

船按照Route执行。

例如：

```
Node15

↓

Node25

↓

Node42

↓

Node87

↓

Node96
```

到达每个RoadNode后：

更新：

* 船当前位置
* 剩余容量
* 剩余能源
* ETA
* Task状态

Graph同步更新。

---

# 十、Rolling Horizon滚动优化

系统不是一次规划到底。

而是：

维护未来5~10个RoadNode。

例如：

```
Node15

Node25

Node42

Node87

Node96

Node108

Node132
```

其中：

前2个节点：

```
Frozen
```

保持不变。

后面的节点：

允许重新优化。

每完成一个节点：

Graph更新。

ALNS重新优化：

剩余Route。

因此：

随着船位置变化。

未来Route不断变优。

---

# 十一、状态检测（Event Detection）

系统持续检测：

## 船状态

检测：

* 船舶故障
* 能源不足
* 长时间停滞

## 任务状态

检测：

* 新增任务
* 任务取消
* 运输异常

若无异常：

继续Rolling Horizon。

若发生异常：

进入Route Repair。

---

# 十二、Route Repair

发生异常后：

不是重新规划全部任务。

而是：

释放受影响Route。

例如：

```
Ship2

Node31

↓

Node45

↓

Node62

↓

Node90
```

故障以后：

释放：

```
Node45

↓

Node62

↓

Node90
```

Graph重新计算：

哪些船最适合继续完成对应任务。

ALNS重新寻找：

最佳插入位置。

例如：

```
Ship5

Node18

↓

Node45

↓

Node62

↓

Node90

↓

Node105
```

完成Repair。

系统继续执行。

---

# 十三、整个系统运行逻辑

```text
① 建立Ship-RoadNode-Task异构图
            │
            ▼
② HGT学习Ship、RoadNode、Task关系
            │
            ▼
③ Graph计算Ship-Task匹配关系
            │
            ▼
④ Graph-guided ALNS生成每艘船RoadNode访问序列(Route)
            │
            ▼
⑤ 船舶按照Route执行运输
            │
            ▼
⑥ 每到达一个RoadNode更新Graph
            │
            ▼
⑦ Rolling Horizon持续优化未来Route
            │
            ▼
⑧ 持续检测船状态和任务状态
            │
      ┌─────┴─────┐
      │           │
     无事件      有事件
      │           │
      ▼           ▼
继续执行      Route Repair
      │           │
      └─────┬─────┘
            ▼
        更新Route
            │
            ▼
         继续执行
```

---

# 十四、各模块职责

| 模块                    | 输入                      | 输出                      | 职责             |
| --------------------- | ----------------------- | ----------------------- | -------------- |
| **异构图(Graph)**        | Ship、RoadNode、Task      | 动态图                     | 实时维护整个运输系统状态   |
| **HGT**               | 异构图                     | Ship-Task匹配评分           | 学习船、任务、路网之间的关系 |
| **Graph-guided ALNS** | Graph + Task + RoadNode | 每艘船的RoadNode访问序列(Route) | 完成初始调度与滚动优化    |
| **Rolling Horizon**   | 当前Route                 | 优化后的未来Route             | 持续优化未执行部分      |
| **Event Detection**   | Ship状态、Task状态           | 是否触发Repair              | 检测异常事件         |
| **Route Repair**      | 异常任务                    | 更新后的Route               | 局部重调度受影响任务     |

---

## 核心思想

> **以 Ship–RoadNode–Task 异构图作为统一状态表达，以 HGT 学习船舶与运输任务之间的匹配关系，以 Graph-guided ALNS 优化每艘船未来的 RoadNode 访问序列，并结合 Rolling Horizon 对未执行部分持续优化，在异常事件发生时采用局部 Route Repair 完成动态重调度。**

系统最终输出每艘船应该依次访问的 RoadNode 序列，直接与已有路网和路径规划模块对接。
